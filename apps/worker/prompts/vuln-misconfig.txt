<methodology>
# White-Box Vulnerability Analysis Procedure: Security Misconfiguration

NOTE ON "FOR ALL": whenever an item begins with "for all …", use the TodoWrite tool to create a separate task for each relevant endpoint/flow that needs to be analyzed.
From `.shannon/deliverables/pre_recon_deliverable.md`, use Section 3 and 6 to help guide your tasks.

## 1) Open Redirect — Phase A: Sink Discovery
- For all endpoints that accept URL/redirect parameters, identify all server-side redirect sinks: framework redirect methods (res.redirect, redirect()), HTTP Location header writes, meta refresh tags, and JavaScript-based redirects (window.location).
- Map each redirect sink to its code location and the parameter(s) that control the target.
**If redirect sinks found → proceed to Phase B.**

## 2) Open Redirect — Phase B: Source-to-Sink Tracing
- For each redirect sink, trace user-controllable input (query parameters, form fields, POST body, headers) to the sink.
- Determine whether the redirect target is fully user-controlled, partially controlled, or derived from a whitelist.
**If user-controllable → proceed to Phase C.**

## 3) Open Redirect — Phase C: Validation Analysis
- For each user-controllable redirect, identify what validation exists: domain whitelist, relative path enforcement, URL scheme restrictions, regex patterns.
- Evaluate whether the validation is sufficient or bypassable (e.g., startsWith('/') bypass via '//' prefix, URL encoding, protocol-relative URLs, domain similarity attacks).
**If validation absent or bypassable → classify:** `Open_Redirect` → **suggested attack:** `open_redirect_protocol_bypass` / `open_redirect_encoding_bypass` / `open_redirect_domain_bypass`.
**If validation sufficient → mark safe in "Secure by Design" section.**

@include(shared/_cross-route-enumeration.txt)

## 4) Open Redirect — Phase D: Impact Assessment
- For each confirmed Open Redirect, assess impact: is this a login/callback redirect (high impact — token leakage, phishing)? A general redirect (medium impact — phishing)? A post-action redirect (lower impact)?
- Note whether tokens or sensitive data appear in URLs during the redirect flow.
**If high impact → confidence: High. If medium/low impact → confidence: Medium.**

## 5) Security Headers
- For all HTML-serving endpoints, verify presence and correctness of:
  - `Content-Security-Policy` (or CSP via meta tag)
  - `Strict-Transport-Security` (max-age >= 31536000, includeSubDomains, preload)
  - `X-Frame-Options` (DENY or SAMEORIGIN — superseded by CSP frame-ancestors but still defense-in-depth)
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy` (strict-origin-when-cross-origin or no-referrer)
  - `Permissions-Policy` (restrict sensitive features)
- For API-only endpoints serving JSON, do NOT flag missing CSP or X-Frame-Options — these are not applicable.
- Check if HSTS max-age is below 1 year (31536000 seconds).
**If missing or weak → classify:** `Missing_Security_Headers` → **suggested attack:** `missing_header_confirmation`.
**If all present and correct → mark safe.**

## 6) CORS Configuration
- For all CORS middleware configuration, check:
  - Dynamic origin reflection (`Access-Control-Allow-Origin` = request Origin) with credentials
  - Wildcard origin (`*`) with credentials
  - Overly permissive allowed methods or headers
  - Null origin allowed
- Wildcard without credentials is safe for public APIs — do NOT flag.
**If dangerous CORS config found → classify:** `CORS_Misconfiguration` → **suggested attack:** `cors_origin_reflection`.
**If safe or public API → mark safe.**

## 7) Cookie Security Flags
- For all cookies set by the application (session, auth, CSRF, tracking), verify:
  - `HttpOnly` flag (prevents JavaScript access)
  - `Secure` flag (HTTPS only)
  - `SameSite` attribute (Lax or Strict)
- Focus on session and authentication cookies.
**If security flags missing → classify:** `Missing_Cookie_Flags` → **suggested attack:** `cookie_flag_confirmation`.
**If all flags present → mark safe.**

## 8) Clickjacking Protection
- For all HTML-serving endpoints, verify frame-busting protection:
  - `Content-Security-Policy: frame-ancestors 'self'` (or more restrictive)
  - OR `X-Frame-Options: DENY` or `SAMEORIGIN`
- If CSP frame-ancestors is set, X-Frame-Options is optional (defense-in-depth only).
**If no frame-busting protection → classify:** `Clickjacking_Vulnerable` → **suggested attack:** `iframe_embedding`.
**If protection present → mark safe.**

## 9) Information Disclosure
- For all error handlers and debug configuration, verify:
  - Stack traces are not returned in production error responses
  - Server version headers are removed or obscured
  - Database error details are sanitized
  - Debug mode is disabled in production
  - Source map files are not accessible
  - Standard error responses do not reveal internal paths or configuration
**If information disclosed → classify:** `Information_Disclosure` → **suggested attack:** `error_disclosure_confirmation`.
**If errors are sanitized → mark safe.**

# Confidence scoring (analysis phase; applies to all checks above)
- **High** — The misconfiguration is directly visible in code/config (e.g., missing middleware registration, absent header setter, redirect with no validation).
- **Medium** — The misconfiguration is strongly indicated but has some uncertainty (e.g., header might be set by upstream proxy, redirect validation might exist in middleware not yet examined).
- **Low** — The misconfiguration is plausible but unverified (e.g., cookie flag might be set by framework default, debug mode might be environment-dependent).

Rule: when uncertain, round down (favor Medium/Low) to minimize false positives.

# Documenting Findings (MANDATORY)
For each check you perform from the list above, you must make a final **verdict**.

- If the verdict is **`vulnerable`**, you must include the finding in your exploitation queue.
- If the verdict is **`safe`**, you **MUST NOT** add the finding to the exploitation queue. These secure components should be documented in the "Secure by Design: Validated Components" section of your final Markdown report.
</methodology>

<false_positives_to_avoid>
- **Flagging missing CSP on API-only endpoints:** CSP is for HTML pages. JSON API endpoints do not need CSP.
- **Flagging CORS wildcard without credentials:** `Access-Control-Allow-Origin: *` without `Access-Control-Allow-Credentials` is a valid public API pattern.
- **Assuming headers from documentation:** Do not treat framework documentation as proof; require actual code/config evidence.
- **Over-testing client-side headers:** Focus on server-side header configuration; client-side headers can be manipulated by the attacker anyway.
- **Confusing development vs production configuration:** Consider environment-specific settings. Flag only if production configuration is confirmed vulnerable.
- **Flagging missing X-Frame-Options when CSP frame-ancestors is present:** CSP frame-ancestors supersedes X-Frame-Options; the latter is defense-in-depth only.
</false_positives_to_avoid>
